## #5 M0 契约补充：图像记录的路径解析与缺图处理【M3 新增，2026-06-12 推送，48h 否决窗口】

- **背景**：M3 = 多模态亮点（你 `dataset-unsafebench` → 乙 `guard-shieldgemma2`）。M0 §3 已定图像真值与输入字段（`label.is_unsafe` + `content.images[].path`），但 path 的**解析基准**与缺图/异常处理 M0 未定义——乙侧实现前需锁定。完整规格见 [`M3_SPEC.md`](M3_SPEC.md) §4.1/§4.2。
- **内容（五点）**：
  1. **路径基准**：`content.images[].path` 为相对路径时，以 **unified JSONL 文件所在目录**为基准解析；绝对路径原样使用。你侧唯一约定动作：图片目录与 unified JSONL 的相对位置保持稳定（建议 `<unified_dir>/images/...`）。
  2. **eligible 判定**：`content.images` 非空且首图 `path` **或** `url` 至少一个非空即 eligible；两者均空 → `skipped.missing_content`（不写输出行）。
  3. **url-only**：只有 `url` 没有本地 `path` 的记录，乙侧不下载（read-only + 网络隔离）→ 记录级 error `image_url_not_supported`。请尽量把图片存盘并填 `path`。
  4. **缺图/坏图可见化**：`path` 文件不存在 → error `image_not_found`；文件损坏/非图像格式 → error `image_decode_error`。三类 error 均照常写输出行、计入 `counts.errors`、进双口径指标——**不静默跳过**，数据缺口在指标里可见。
  5. **多图记录**：M3 只评首图（UnsafeBench 为单图，此为防御性约定）；多图记录会被 warning 计数（run_metadata + 行级标注），**不算错误**。
- **为什么需要**：#1 的布局补充只覆盖了 Guard **输出**侧；图像**输入**侧的路径语义是甲乙文件交换的最后一处未定义点，M2 多 Guard 对比与 M3 跨模态评测都建立在它上面。
- **影响面**：只增不改——你已有字段语义、checker 验收口径零变化；text 侧完全不受影响。
- **请你做**：无异议**无需任何动作**（默认接受，自本条推送起 48h，约至 06-14）；有异议回"#5 + 理由"。若 UnsafeBench 实际落盘布局与 ① 不一致，直接回实际布局即可——乙侧解析基准是单函数修改，改起来很快。
