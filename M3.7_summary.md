## 2. 真实数字

### 2.1 评测设置
- 子集 = `out_m3_real` int8 NaN error 行的**前 100 个 id**（共 395；19.4%），切片 `e3_nan100.subset.jsonl`（gitignored）。
- 命令 `--nan-fallback cpu --device auto --timeout-s 300 --model-id Nozim6690/...`（offline，HF_HUB_CACHE=E）；**~2.4h**（cpu bf16 ~74–94s/图）。
- 8GB 上 `auto` 必然降级 cpu（int8+fp16 共存 OOM，dead-mode 一次缓存；机制 unit-tested）→ 评测用 cpu 直跑求干净，恢复分与 auto 等价。

### 2.2 恢复结果
| | 值 |
|---|---|
| 恢复数 | **100/100**（`nan_fallback_recovered=100`，0 error，全部 `quant=int8->cpu-bf16`） |
| 子集覆盖率 before→after | **0% → 100%** |
| 全量覆盖率投影 | int8 1642/2037=**80.6%** → 全 395 回退后 **~100%** |
| 子集真值分布 | 28 unsafe / 72 safe |

### 2.3 恢复子集判别质量（cpu-bf16 @ 0.30，vs UnsafeBench 真值，n=100）
| Acc | Recall | FPR | Macro-F1 | AUROC | coverage |
|---|---|---|---|---|---|
| 0.790 | 0.393 | 0.056 | 0.689 | **0.675** | 1.000 |

- 恢复分**是 truth-grade**：AUROC **0.675** 贴近 CPU bf16 真值水平（全量 0.719）、远高于 int8 下界 0.613——证明回退找回的是有用判定，不是垃圾。
- 这 100 张是 int8 **连分都给不出**的更难子集（NaN），故其绝对指标不与全量直接可比；关键是它们从"全丢"变成"truth-grade 已答"。

## 3. 诚实口径（关键，写进报告）
- **E3 = 覆盖率/鲁棒性赢点，不是 int8 质量修复**：全量在 E3 后是**混精度**——80.6% 仍是漂移的 int8（notes §6.1a：连非-NaN 分都只约一半可信）+ 19.4% truth-grade bf16。多数 int8 的判别漂移**未解**。
- **阈值口径**：0.30 是在 int8 分上校准的，这里套在 bf16 分上；AUROC（阈值无关）0.675 表明排序合理，但 0.30 对 bf16 非最优——如实标注。
- **结构上限仍未解、仅定位**：三策略天花板 / judge 对抗 / int8 量化漂移（E3 只补覆盖，不动这三者）。
- `--nan-fallback none`（默认）逐位复现旧 error-row 行为；cpu 回退 ~74–94s/图，全覆盖跑须给足 `--timeout-s`。

## 4. 工程过程（评测抓出真 bug）
- 首跑 0/100：cpu forward ~94s 撞上 120s 软超时（且当时迁移 cp 并发），dead-mode 把**瞬态超时**当确定性失败 → 杀 cpu → 级联 98 失败。
- **T1c 修复**：区分确定性（OOM/加载失败 → mode 死亡）与瞬态（`TimeoutError` → 只跳过当图）；加回归测试。重跑 100/100。
- 套件：**sg2 110 绿**（T1/T1b/T1c/T2 共 +10 测试，含 cpu/gpu/auto-OOM→cpu/瞬态超时不级联/也-NaN→error/none 不回退/run_metadata 落地）；llama 105 不变；**metrics.py 未动**（恢复行=普通 answered 行，受控复制零改）。
